01 — Por que a análise de PR falha em produção

Causa-raiz e o que precisa ser feito para o workflow rodar. Achados PA-01 a PA-05.

Modelo mental: o que o pipeline realmente faz

Tanto a análise manual (POST /api/repositories/{id}/pull-requests/{n}/analyze) quanto o re-run (POST /api/analysis-runs/{id}/execute) chamam analysis_execution_service.execute_analysis_run, que na própria thread da requisição:

  1. Gera token de instalação do GitHub App.
  2. Para cobertura: baixa o repositório no base_sha e no head_sha, roda install_command e test_command em cada um, e parseia o relatório.
  3. Para segurança: baixa o head, roda semgrep, detect-secrets e (em Python) bandit e pip-audit via subprocess.
  4. Para débito técnico: baixa o head de novo e parseia arquivos.
  5. Gera review de IA (OpenAI) e monta o relatório final.

O deploy de produção é serverless na Vercel:

// vercel.json
"backend": { "entrypoint": "backend/main.py", "routePrefix": "/server", "maxDuration": 300 }
O backend/Dockerfile que instala git, golang-go, nodejs e npm só é usado pelo docker-compose local. A Vercel ignora esse Dockerfile e empacota a função com o builder Python — sem esses binários e com filesystem somente-leitura (exceto /tmp).

PA-01 — O runtime de produção não tem o toolchain nem filesystem gravável que o runner exige Crítico

Local
backend/app/services/runner_service.py (run_command, linha 249) · coverage_gate._run_revision_coverage · security_gate._scanner_commands
Mecanismo

Os gates fazem subprocess.run(command, shell=True) com comandos que pressupõem um ambiente de CI completo:

  • Cobertura: install_command padrão = pip install -r requirements.txt (grava em site-packages, que é somente-leitura na Vercel → falha). Para repos TS/JS/Go, exige npm/go que não existem na função.
  • Segurança: semgrep --config=auto baixa regras da internet e precisa do binário; detect-secrets/bandit/pip-audit são entry-points que podem não estar no PATH da função.
Cenário concreto

Repositório Python padrão → cobertura roda pip install -r requirements.txt → OSError: [Errno 30] Read-only file system → RunnerError("Coverage install command failed.") → o run termina em ERROR com "Gate execution failed". Repositório Node/Go → install/test retornam 127 command not found → mesmo desfecho. Nenhuma análise consegue passar do gate de cobertura em produção.

Evidência histórica

O commit df6cd5c "fix: fetch PR analysis revisions without git binary" já mostra que a equipe bateu nessa parede com o git. A mesma ausência de binário/FS afeta pip/npm/go/scanners.

Correção

Mover a execução para um runtime que tem o toolchain e disco gravável (worker container dedicado, ou GitHub Actions como executor). Ver fixes/02.

PA-02 — O pipeline inteiro roda síncrono dentro da requisição HTTP Crítico

Local
backend/app/api/routes_repositories.py:97 e routes_analysis.py:63 → execute_analysis_run 
if run.status == AnalysisRunStatus.PENDING:
    return analysis_execution_service.execute_analysis_run(db, run.id)  # bloqueia a request
Mecanismo

Clonar + instalar dependências + rodar a suíte de testes (duas vezes: base e head) + scanners + chamada à OpenAI levam, no caso real, de dezenas de segundos a vários minutos. Tudo isso acontece antes de a rota responder. A função serverless tem maxDuration=300s e o cliente do frontend (fetch) não tem timeout estendido.

Cenário concreto

Usuário clica "Analisar PR" → a request fica pendurada → a função estoura maxDuration → 504 / FUNCTION_INVOCATION_TIMEOUT. O AnalysisRun fica preso em RUNNING (ver PA-06) e o frontend mostra erro genérico.

Correção

A rota deve apenas enfileirar o run e responder 202 imediatamente; um worker assíncrono executa o pipeline; o frontend faz polling do status. Ver fixes/01.

PA-03 — command_timeout_seconds (600s) é maior que maxDuration (300s) Alto

Local
backend/app/core/config.py:37 (command_timeout_seconds: int = 600) vs vercel.json:10 (maxDuration: 300)
Mecanismo

Um único comando pode ser autorizado a rodar por 600s, mas a função é morta aos 300s. Como o pipeline encadeia 4+ comandos (install+test para base e head), o tempo total quase sempre ultrapassa 300s. O processo é morto no meio de um subprocess, sem cair em nenhum except — o run nunca é marcado como ERROR e fica preso em RUNNING.

Correção

Em qualquer arquitetura, o timeout efetivo de toda a análise deve ser menor que o limite do runtime que a hospeda. Com worker dedicado, alinhar command_timeout_seconds e um teto global do pipeline. Ver fixes/02.

PA-04 — Execução via BackgroundTasks do webhook é morta após a resposta em serverless Alto

Local
backend/app/services/github_webhook_service.py:108-112 
if created_new:
    if background_tasks is not None:
        background_tasks.add_task(_execute_run_by_id, run.id)   # roda DEPOIS da resposta
    else:
        analysis_execution_service.execute_analysis_run(db, run.id)
Mecanismo

O BackgroundTasks do Starlette roda depois de a resposta ser enviada, mas dentro da mesma invocação. Em serverless, a plataforma congela/encerra a função assim que a resposta HTTP é entregue, então a task de análise iniciada é interrompida antes de terminar (ou nem começa). O webhook responde 202 alegremente e o run fica em PENDING/RUNNING para sempre.

Cenário concreto

Pull Request aberto → GitHub envia pull_request.opened → backend cria o run e agenda a task → responde 202 → função encerra → análise nunca executa. Para o usuário, "criei o PR mas nada aconteceu".

Correção

Mesma solução de PA-02: enfileirar em fila/worker durável, nunca depender de trabalho pós-resposta no serverless. Ver fixes/01.

PA-05 — Download do archive bate direto no codeload com Bearer token → falha em repos privados Alto (plausível)

Local
backend/app/services/runner_service.py:142-179 (download_repository_archive) 
archive_url = f"https://codeload.github.com/{owner}/{name}/tar.gz/{revision}"
request = urllib.request.Request(archive_url, headers=_github_archive_headers(token))
# headers["Authorization"] = f"Bearer {token}"
Mecanismo

O caminho documentado/estável para baixar tarball com token de App é GET https://api.github.com/repos/{owner}/{repo}/tarball/{ref}, que responde 302 para uma URL codeload assinada (token na query, expira em segundos). Bater direto no host codeload com header Authorization: Bearer não é o fluxo suportado para conteúdo privado e frequentemente retorna 404 (o GitHub trata repo privado como inexistente quando a auth não é aceita naquele host).

Cenário concreto

Repositório privado → HTTP 404 Not Found capturado como HTTPError → RunnerError("Repository checkout failed...") → todos os gates que precisam de checkout falham. Repositórios públicos funcionam, o que pode mascarar o bug em testes manuais.

Como confirmar

Tentar a análise em um repositório privado e inspecionar coverage_result_json.commands[].stderr — se aparecer HTTP 404 no download, é isto.

Correção

Usar o endpoint /repos/{owner}/{repo}/tarball/{ref} da API e seguir o redirect (sem reenviar o header Authorization para o host assinado). Ver fixes/04.

Resumo: o que destrava o workflow

#AçãoResolve
1Tirar o pipeline da request: enfileirar + worker durável + polling de statusPA-02, PA-04
2Executar a análise num runtime com toolchain e disco gravável (worker container ou GitHub Actions), não na função serverlessPA-01, PA-03
3Baixar o tarball via API /tarball/{ref} com redirectPA-05
4Envolver o pipeline em try/finally que sempre fecha o estado do runPA-06 (ver relatório 02)
