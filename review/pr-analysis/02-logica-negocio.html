PA-06 — Exceção após status=RUNNING deixa o run preso para sempre Alto

Local
backend/app/services/analysis_execution_service.py:19-124
Mecanismo

Em execute_analysis_run, a linha 39 marca status = RUNNING e commita (linha 51). A partir daí não há try/finally. Se qualquer passo subsequente levantar uma exceção que os gates não capturam internamente — por exemplo quality_agent (captura tudo, ok), mas também report_service.build_final_report, um db.commit() que estoure, ou o kill por timeout de PA-03 — o run permanece em RUNNING. O guard de entrada só permite re-execução de PENDING ou ERROR:

if run.status not in (AnalysisRunStatus.PENDING, AnalysisRunStatus.ERROR):
    raise AppError(409, "analysis_run_not_pending", ...)
Cenário concreto

Timeout da Vercel (PA-03) mata a função no meio de um subprocess → o except dos gates nunca roda → o run fica RUNNING permanentemente → o botão "re-run" (commit 5bdd089, que só liberou ERROR) retorna 409. O usuário não consegue nem reanalisar nem limpar. Esse é exatamente o "trava em produção" relatado.

Correção

Envolver todo o corpo em try/except Exception que chama _finish_with_error, e/ou permitir re-execução de runs RUNNING antigos (com guarda de "stale"). Ver fixes/04.

PA-08 — Cobertura instala e roda a suíte inteira para base e head Alto

Local
backend/app/services/gates/coverage_gate.py:49-50 
base_report = _run_revision_coverage(workspace, base_sha, coverage_config)
head_report = _run_revision_coverage(workspace, head_sha, coverage_config)

Cada _run_revision_coverage faz checkout + install_command + test_command + parse. São 4 comandos pesados por análise (2 instalações completas + 2 suítes de teste completas).

Impacto de negócio

Além de dobrar custo/tempo (principal motor dos timeouts de PA-02/PA-03), recalcular a cobertura total da base a cada análise raramente agrega valor: a métrica que decide o gate de "changed files" usa só o head. A cobertura da base poderia vir de um snapshot/cache do último run no merge-base, não de uma re-execução completa.

Correção

Rodar somente o head por padrão; tornar a medição da base opcional (só quando max_coverage_drop está configurado) e/ou cachear por SHA. Ver fixes/05.

PA-07 — Arquivo alterado ausente do relatório conta como 0% de cobertura Médio

Local
backend/app/services/gates/coverage_gate.py:107-116 
for filename in source_files:
    coverage = normalized_report.get(_normalize_path(filename))
    if coverage is None:
        percentages.append(0)      # ausente = 0%
        continue
    percentages.append(coverage.percentage)
percentage = round(sum(percentages) / len(percentages), 2)   # média simples
Problema de regra

Dois defeitos combinados: (1) um arquivo-fonte alterado que não aparece no relatório de cobertura (porque não foi exercitado, ou porque o path não casou — ver PA-09) é tratado como 0%, derrubando a média; (2) a média é simples por arquivo, não ponderada por linhas, então um arquivo pequeno pesa igual a um grande. O resultado é "changed files coverage" instável e frequentemente injusto → falha de gate falsa.

Cenário concreto

PR adiciona migrations/0005.py (sem testes, por design) + altera core.py (95% coberto). Média = (0 + 95)/2 = 47,5% → abaixo do mínimo → PR reprovado por engano.

Correção

Distinguir "arquivo sem dados de cobertura" de "0% coberto" (excluir/ignorar, ou exigir match explícito), e ponderar por linhas cobertas/total. Ver fixes/05.

PA-09 — Mismatch de paths entre GitHub e o relatório de cobertura Médio

Local
coverage_gate.py:108 (_normalize_path, linha 245) e is_source_file
Mecanismo

Os paths dos arquivos alterados vêm da API do GitHub, sempre relativos à raiz do repositório (src/app/core.py). Os paths do relatório de cobertura dependem da ferramenta: Cobertura emite filename relativo ao diretório de execução (pode ser core.py ou app/core.py dependendo do working_directory), lcov usa o path do source-root, go usa o import-path do módulo. _normalize_path só troca \ por / e tira ./ — não reconcilia prefixos. Quando working_directory != ".", o relatório é gerado dentro do subdiretório e os paths quase nunca casam com os do GitHub.

Impacto

Quase todo arquivo alterado vira "ausente" → 0% (ver PA-07) → cobertura de arquivos alterados sistematicamente subestimada em monorepos / projetos com working_directory.

Correção

Normalizar ambos os lados pelo working_directory e casar por sufixo de path (basename + segmentos) com fallback. Ver fixes/05.

PA-10 — Analisador de "função longa" para linguagens de chave é um contador ingênuo de {} Médio

Local
backend/app/services/gates/technical_debt_gate.py:187-222 (analyze_brace_language_file) 
brace_depth = line.count("{") - line.count("}")
while brace_depth > 0 and end_index + 1 < len(lines):
    end_index += 1
    brace_depth += lines[end_index].count("{") - lines[end_index].count("}")
Problema

Conta { e } textualmente, incluindo os que aparecem em strings, template literals, regex e comentários (const re = /\{.*\}/;, `${x}`, // usa { }). Em TS/JS isso é comum, então o cálculo de "linhas da função" fica errado — ora superestima, ora nunca fecha e engole o resto do arquivo como uma função gigante. O finding é blocking=True, ou seja, reprova o PR com base em uma medição incorreta.

Cenário concreto

Arquivo TS com uma função de 5 linhas contendo const s = "}"; → a contagem de chave fecha cedo ou tarde demais → "Function exceeds line limit" falso → PR bloqueado.

Correção

Ou usar um parser real (ex.: tree-sitter) para linguagens de chave, ou rebaixar esse finding de blocking para informativo até existir parsing confiável. Ver fixes/05 (seção "débito técnico").

Observações adicionais de negócio (menores)

  • Reuso por (repo, pr_number, head_sha): um run já concluído é reusado em vez de reanalisado — correto, mas significa que mudanças de configuração do quality gate (ex.: baixar min_total_coverage) não reavaliam um head já analisado. Documentar esse comportamento ou permitir "forçar reanálise".
  • Severidade do semgrep: _normalize_severity mapeia "error" → HIGH e "warning" → HIGH (mesma severidade), achatando a distinção entre erro e aviso do semgrep.
  • _blocking_severities([]) retorna conjunto vazio → se um admin configurar security_fail_on = [], nenhuma vulnerabilidade bloqueia, mas o gate ainda reporta "pass". Pode ser intencional, mas vale um aviso na UI.
