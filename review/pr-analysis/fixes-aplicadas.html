Fix 03 — Vazamento de segredos e redação de saída

PA-11PA-13PA-14

Problema

O subprocess dos gates herdava todo o ambiente menos alguns prefixos (blocklist). SESSION_SECRET e TOKEN_ENCRYPTION_KEY não estavam na lista e vazavam para código não-confiável. stdout/stderr eram persistidos no snapshot (e podiam ir para comentários no GitHub) sem redação.

O que mudou

Blocklist → allowlist em _safe_env: o processo só vê o mínimo do toolchain (PATH, HOME, GOPATH, NODE_ENV…). Qualquer segredo novo no .env passa a ser seguro por padrão. Adicionado redact_secrets() (tokens ghs_/gho_/github_pat_, credencial x-access-token:…@ e blocos de chave privada) aplicado em CommandResult.to_snapshot.

Arquivos
  • backend/app/services/runner_service.py

TDD

Redtest_runner_secrets.py: _safe_env ainda continha SESSION_SECRET; redact_secrets nem existia.
GreenAllowlist + redact_secrets; 7 testes passam (não-vazamento, PATH default, mascaramento de tokens/chave).
Resultado esperado: nenhum segredo da app entra no ambiente do scanner e a saída persistida nunca contém tokens/chaves em texto claro.
PA-12 (isolar a execução de código não-confiável em sandbox/Actions) é decisão de arquitetura — a allowlist reduz o vazamento, mas o sandbox em si fica fora do escopo de código testável.

Fix 01 — Execução assíncrona com fila durável

PA-02PA-04

Problema

O pipeline pesado rodava dentro da requisição HTTP (timeout/504) e o BackgroundTasks morre em runtime serverless. Resultado: "erro em produção".

O que mudou

As rotas agora só enfileiram e respondem 202 Accepted; um worker durável consome a fila e executa o pipeline. Fila no próprio Postgres com FOR UPDATE SKIP LOCKED (sem Redis/Celery).

POST …/analyze ─▶ cria AnalysisRun (PENDING) ─▶ enqueue ─▶ 202 {run_id}
                                                  │
                       worker (container) ◀───────┘ executa o pipeline
frontend ─▶ GET /analysis-runs/{id}  (polling até completed | error)
Arquivos
  • app/models/analysis_job.py + migração …_0009_add_analysis_jobs.py
  • app/services/analysis_queue.py (enqueue/claim_next)
  • app/worker.py (process_next_job/run_forever)
  • routes_repositories.py, routes_analysis.py, webhook → enqueue + 202
  • frontend/.../AnalysisDetailPage.tsx (polling de 3s enquanto pending/running)

TDD

RedFila/worker inexistentes; testes de rota ainda esperavam 200 + run executado.
Greentest_analysis_queue.py (enqueue idempotente, claim único), test_worker.py (claim+execute, erro engolido) e contratos de rota assíncronos passam.
Resultado esperado: a request responde na hora; a análise roda fora dela e o estado evolui PENDING → RUNNING → COMPLETED/ERROR via polling.
Os testes do pipeline foram repontados para chamar execute_analysis_run diretamente (a rota deixou de executar de forma síncrona — passou a só enfileirar).

Fix 02 — Runtime com toolchain e teto de tempo

PA-01PA-03

Problema

A função serverless não tem git/npm/go/scanners nem disco gravável, e command_timeout_seconds podia exceder o teto do runtime.

O que mudou

Opção A (worker em container) adotada: serviço worker no docker-compose.yml usando o backend/Dockerfile (que já tem o toolchain), com smoke-test dos scanners no build. Introduzido o Runner Protocol para trocar a implementação de execução sem reescrever os gates. Adicionado analysis_total_timeout_seconds (900s) e um deadline global aplicado antes de cada gate.

Arquivos
  • app/services/gates/runner_protocol.py (Protocol runtime_checkable)
  • app/core/config.py (analysis_total_timeout_seconds)
  • app/services/analysis_execution_service.py (deadline por gate)
  • backend/Dockerfile (smoke-test), docker-compose.yml (serviço worker)

TDD

Redrunner_protocol inexistente; orçamento total de tempo não respeitado.
Greentest_runner_protocol.py (RunnerWorkspace satisfaz o Protocol; default 900s) e teste que aborta a análise quando o orçamento estoura passam.
Resultado esperado: a análise roda num runtime com toolchain e disco gravável, e nunca ultrapassa o teto global — vira ERROR controlado em vez de travar.

Fix 04 — Run preso e download de repo privado

PA-05PA-06PA-15

Problema

Exceção após status = RUNNING deixava o run preso (guard só aceitava PENDING/ERROR). Download direto do codeload com Bearer dava 404 em repositório privado. create_or_reuse_* tinha código duplicado.

O que mudou

Pipeline envolvido em try/except: qualquer exceção vira ERROR via _finish_with_error. Runs RUNNING antigos (>30 min) podem ser reexecutados (recuperação de stale). Download passou para o endpoint oficial /repos/{owner}/{repo}/tarball/{ref} e o Authorization é removido ao seguir o redirect assinado do codeload. Builder comum _build_pending_run + _create_or_reuse.

Arquivos
  • app/services/analysis_execution_service.py (try/except + stale)
  • app/services/runner_service.py (_archive_url, _NoAuthOnRedirect)
  • app/services/analysis_service.py (dedup do builder)

TDD

RedExceção no gate propagava (run ficava RUNNING); URL ainda era codeload…tar.gz; _build_pending_run não existia.
GreenTestes de stale-rerun, "exceção → ERROR", URL api…/tarball, redirect sem Authorization e builder compartilhado passam.
Resultado esperado: nenhum run fica preso em RUNNING; o tarball de repositório privado baixa via URL assinada sem vazar o Bearer para o host de download.

Fix 05 — Lógica de cobertura e débito técnico

PA-07PA-08PA-09PA-10

Problema

A base de cobertura era sempre medida (2× custo). Arquivo alterado sem dado de cobertura contava como 0% e a média era simples (não ponderada). lstrip("./") corrompia paths como .github/…. Função "longa" em JS/TS/Go bloqueava o PR com heurística de contagem de chaves não-confiável.

O que mudou

  • PA-08: base só é medida quando max_coverage_drop está ativo; sem base, a regra de drop não dispara.
  • PA-07: arquivo sem dado vira unmatched (aviso), não 0%; média ponderada por linha; se nenhum arquivo tem dado, retorna None.
  • PA-09: removeprefix("./"), casamento tolerante a working_directory e fallback por sufixo.
  • PA-10: finding de função longa em linguagem de chave vira blocking=False (LOW) — visível, mas não reprova.
Arquivos
  • app/services/gates/coverage_gate.py
  • app/services/gates/technical_debt_gate.py

TDD

RedArquivo ausente dava 40% (média com 0); .github/… virava github/…; base sempre medida; brace finding com blocking=True.
GreenTestes de ponderação por linha, unmatched, prefixo de working_directory, fallback por sufixo, base ignorada sem policy e brace não-bloqueante passam.
Resultado esperado: menos falsos positivos/negativos de cobertura, paths casados corretamente, metade do custo quando não há regra de drop, e nenhum PR reprovado por heurística de chaves.
Gerado a partir da implementação por TDD dos planos em review/pr-analysis/fixes/. Suíte do backend: 135 testes passando (pytest); frontend verificado com tsc -b. Decisão de arquitetura registrada: worker em container (Opção A); isolamento por sandbox (PA-12) e delegação ao GitHub Actions (Opção B) ficam como evolução.
