# PR Quality Gate Dashboard

## 1. Visão geral

O **PR Quality Gate Dashboard** é uma aplicação web para analisar Pull Requests do GitHub usando critérios objetivos e inteligência artificial. O objetivo do projeto é permitir que um time ou desenvolvedor conecte um repositório GitHub, configure regras de qualidade e execute uma análise completa de um PR antes do merge.

O sistema verifica três pilares principais:

1. **Coverage de testes**
2. **Segurança**
3. **Dívida técnica**

Ao final da análise, o dashboard mostra uma decisão simples:

```text
PASS — o PR passou no quality gate

ou

FAIL — o PR deve ser corrigido antes do merge
```

Além disso, o sistema pode comentar automaticamente no Pull Request e publicar um status no GitHub, como:

```text
ai-quality-gate: success
ai-quality-gate: failure
```

A primeira versão do projeto será simples, manual e direta. O usuário acessa o dashboard, conecta um repositório, escolhe um PR e clica em **Analisar PR**.

---

## 2. Objetivo do projeto

O objetivo principal é criar uma plataforma web que ajude a responder a seguinte pergunta:

```text
Este Pull Request está seguro, testado e tecnicamente saudável o suficiente para ser aprovado?
```

O sistema deve evitar que PRs problemáticos avancem sem revisão adequada, especialmente quando eles:

* Reduzem a cobertura de testes.
* Adicionam código crítico sem testes.
* Introduzem risco de segurança.
* Criam dívida técnica relevante.
* Adicionam funções muito grandes ou complexas.
* Criam endpoints sem autenticação ou autorização clara.
* Adicionam TODOs ou soluções temporárias em código novo.
* Alteram partes sensíveis do sistema sem cuidado.

## 3. Stack do projeto

A stack escolhida para o projeto será:

```text
Backend: FastAPI
Frontend: React
Banco de dados: PostgreSQL
ORM: SQLAlchemy
Migrations: Alembic
Agente de IA: LangChain
Containerização: Docker Compose
```

### 3.1 FastAPI

O **FastAPI** será responsável pela API principal da aplicação. Ele irá expor endpoints para:

* Autenticação e conexão com GitHub.
* Listagem de repositórios.
* Listagem de Pull Requests.
* Configuração de quality gates.
* Execução de análises.
* Consulta de resultados.
* Histórico de análises.
* Publicação de comentários/status no GitHub.

### 3.2 React

O **React** será usado para criar o dashboard web. Ele será responsável por:

* Exibir repositórios conectados.
* Exibir Pull Requests abertos.
* Permitir configuração das regras de quality gate.
* Mostrar análises em andamento.
* Mostrar resultados detalhados.
* Exibir histórico e métricas do projeto.

### 3.3 PostgreSQL

O **PostgreSQL** será usado para armazenar:

* Usuários.
* Instalações/conexões com GitHub.
* Repositórios conectados.
* Configurações por repositório.
* Execuções de análise.
* Findings encontrados.
* Relatórios finais.

### 3.4 SQLAlchemy

O **SQLAlchemy** será usado como ORM para mapear as tabelas do banco para classes Python.

Ele será responsável por facilitar:

* Criação de registros.
* Consultas.
* Relacionamentos.
* Atualizações.
* Filtros.
* Transações.

### 3.5 Alembic

O **Alembic** será usado para controlar migrations do banco de dados.

Toda alteração estrutural no banco deverá ser versionada com migration, por exemplo:

* Criar tabela de repositórios.
* Criar tabela de análises.
* Adicionar coluna de status.
* Alterar estrutura de configurações.

### 3.6 LangChain

O **LangChain** será usado para construir o agente de análise inteligente.

Ele será responsável por interpretar os dados coletados do PR, como:

* Diff.
* Arquivos alterados.
* Resultado de coverage.
* Resultado de scanners de segurança.
* Métricas de dívida técnica.

O LangChain deve gerar uma análise estruturada com:

* Decisão final.
* Score.
* Riscos encontrados.
* Motivos bloqueantes.
* Sugestões de correção.
* Resumo técnico do PR.

### 3.7 Docker Compose

O **Docker Compose** será usado para rodar o ambiente local com múltiplos serviços:

* Backend FastAPI.
* Frontend React.
* PostgreSQL.
* Serviço de análise, caso seja separado futuramente.

No MVP, o backend pode executar a análise diretamente. Em versões futuras, o runner pode ser separado em um worker.

## 4. Proposta do produto

O produto será um dashboard de análise de Pull Requests.

Fluxo principal:

```text
Usuário acessa o dashboard
→ conecta um repositório GitHub
→ configura as regras do quality gate
→ escolhe um Pull Request
→ clica em "Analisar PR"
→ sistema executa análise
→ dashboard mostra PASS ou FAIL
→ sistema pode comentar no PR e publicar status no GitHub
```

O foco inicial não é substituir CI/CD, nem GitHub Actions. O foco é criar um **revisor técnico inteligente**, executado a partir de um dashboard.

## 7. Fluxo de análise do PR

O fluxo principal de análise será:

```text
1. Usuário clica em "Analisar PR"
2. Backend cria um analysis_run com status running
3. Backend busca dados do PR no GitHub
4. Backend clona o repositório em workspace temporário
5. Backend roda análise de coverage
6. Backend roda análise de segurança
7. Backend roda análise de dívida técnica
8. Backend envia dados consolidados para o agente LangChain
9. Agente gera decisão estruturada
10. Backend aplica regras finais objetivas
11. Resultado é salvo no banco
12. Dashboard exibe o resultado
13. Opcionalmente, sistema comenta no PR
14. Opcionalmente, sistema publica status no commit do PR
```

## 8. Coverage Gate

O **Coverage Gate** é responsável por verificar se o PR mantém ou melhora a cobertura de testes.

Ele deve comparar:

```text
coverage da branch base
coverage da branch do PR
```

Exemplo:

```text
Base: 84.2%
PR: 82.9%
Drop: -1.3%
```

Regras recomendadas:

```text
FAIL se coverage total ficar abaixo do mínimo configurado.
FAIL se coverage cair mais que o permitido.
FAIL se arquivos alterados tiverem coverage abaixo do mínimo.
WARN se o PR adiciona código novo sem adicionar testes.
```

Configuração inicial:

```json
{
  "min_total": 80,
  "max_drop": 0,
  "min_changed_files": 75
}
```

Resultado esperado:

```json
{
  "status": "fail",
  "base_coverage": 84.2,
  "pr_coverage": 82.9,
  "coverage_drop": 1.3,
  "changed_files_coverage": 61.5,
  "blocking_reasons": [
    "Coverage caiu de 84.2% para 82.9%",
    "Coverage dos arquivos alterados está abaixo de 75%"
  ]
}
```

## 9. Security Gate

O **Security Gate** é responsável por detectar riscos de segurança no PR.

Ele deve usar uma combinação de:

```text
scanners determinísticos
+
revisão inteligente com LangChain
```

Scanners iniciais recomendados para projetos Python:

```text
bandit
detect-secrets
semgrep
pip-audit
```

O LangChain deve revisar o diff buscando riscos como:

* Endpoint novo sem autenticação.
* Autorização fraca.
* Falta de validação de input.
* SQL injection.
* SSRF.
* IDOR.
* Path traversal.
* Exposição de token.
* Dados sensíveis em logs.
* CORS aberto demais.
* Bypass de autenticação.
* Uso inseguro de variáveis de ambiente.
* Permissões excessivas.

Regras recomendadas:

```text
FAIL se houver severidade critical.
FAIL se houver severidade high.
WARN se houver severidade medium.
PASS se não houver bloqueadores.
```

Resultado esperado:

```json
{
  "status": "fail",
  "critical": 0,
  "high": 1,
  "medium": 2,
  "low": 3,
  "blocking_reasons": [
    "Novo endpoint /admin/export não possui autorização clara de admin"
  ],
  "warnings": [
    "Payload recebido no endpoint não possui validação forte",
    "Log pode expor e-mail do usuário"
  ]
}
```

## 10. Technical Debt Gate

O **Technical Debt Gate** é responsável por detectar se o PR adiciona dívida técnica nova.

Ele deve verificar:

* Funções grandes demais.
* Arquivos grandes demais.
* Complexidade alta.
* TODO/FIXME novo.
* Uso excessivo de `except Exception`.
* Código duplicado.
* Nomes ruins.
* Responsabilidades misturadas.
* Falta de tipagem em código novo.
* Código difícil de testar.
* Baixo isolamento de regras de negócio.
* Acoplamento excessivo.

Ferramentas possíveis:

```text
radon
ruff
mypy
análise própria via AST
```

A IA deve complementar a análise identificando problemas arquiteturais mais subjetivos.

Regras recomendadas:

```text
FAIL se uma função nova passar do limite de linhas.
FAIL se complexidade passar do limite.
FAIL se houver TODO/FIXME novo em fluxo principal.
FAIL se o agente encontrar blocker arquitetural.
WARN para sugestões de melhoria não bloqueantes.
```

Configuração inicial:

```json
{
  "max_function_lines": 80,
  "max_complexity": 10,
  "fail_on_new_todo": true
}
```

Resultado esperado:

```json
{
  "status": "fail",
  "score": 68,
  "blocking_reasons": [
    "A função process_import tem 142 linhas",
    "O PR adiciona TODO em fluxo principal",
    "O serviço mistura validação, transformação e persistência"
  ],
  "suggestions": [
    "Separar validação em uma classe própria",
    "Criar testes unitários para transformação dos dados",
    "Mover persistência para um repository/service dedicado"
  ]
}
```

## 11. Agente LangChain

O agente LangChain será responsável por consolidar as análises e gerar uma resposta estruturada.

O agente não deve substituir as regras objetivas. Ele deve atuar como uma camada de interpretação e revisão técnica.

Entrada do agente:

```text
dados do PR
arquivos alterados
diff
resultado do coverage gate
resultado do security gate
resultado do technical debt gate
configuração do repositório
```

Saída esperada:

```json
{
  "decision": "fail",
  "score": 72,
  "summary": "O PR adiciona uma funcionalidade relevante, mas reduz coverage e cria dívida técnica significativa.",
  "coverage_status": "fail",
  "security_status": "pass",
  "technical_debt_status": "fail",
  "blocking_reasons": [
    "Coverage caiu 1.3%",
    "Função nova excede limite de tamanho",
    "Fluxo principal não possui teste para payload inválido"
  ],
  "suggestions": [
    "Adicionar testes para payload inválido",
    "Refatorar função process_import",
    "Separar validação, transformação e persistência"
  ]
}
```

O backend deve aplicar uma regra final objetiva:

```text
Se coverage falhou → FAIL
Se segurança tem high ou critical → FAIL
Se dívida técnica tem blocker → FAIL
Se score final for menor que o mínimo → FAIL
Caso contrário → PASS
```

Essa abordagem evita que a IA aprove um PR que falhou em uma regra objetiva importante.

```markdown
- Base: 84.2%

- Drop: -1.3%
- Arquivos alterados: 61.5%

### Segurança

- Critical: 0
- High: 0
- Medium: 2

Resultado: ✅ Aprovado sem bloqueadores críticos

### Dívida técnica

Resultado: ❌ Reprovado

Bloqueadores:

1. `src/users/service.py` adicionou uma função com 142 linhas.
2. O PR adicionou `TODO` em fluxo principal.
3. O endpoint novo não possui teste para payload inválido.

### Recomendações

1. Separar validação, transformação e persistência.
2. Adicionar testes para payload inválido.
3. Reduzir tamanho da função principal.
```

## 19. Riscos técnicos

### Execução de código não confiável

O sistema pode executar testes de um PR que contém código malicioso.

Mitigação:

* Usar container isolado.
* Usar timeout.
* Limitar recursos.
* Evitar montar arquivos sensíveis.
* Não expor secrets no ambiente do runner.

### Falso positivo da IA

A IA pode reprovar um PR por um motivo exagerado.

Mitigação:

* Separar bloqueadores de sugestões.
* Usar regras objetivas.
* Permitir revisão humana.
* Salvar histórico para ajustar prompts.

### Falso negativo da IA

A IA pode deixar passar um problema real.

Mitigação:

* Usar scanners determinísticos.
* Regras objetivas para segurança.
* Scanners como Bandit, Semgrep e detect-secrets.
* Melhorar prompts com casos reais.

### Coverage lento

Rodar testes pode demorar muito.

Mitigação:

* Timeout.
* Configuração por repositório.
* Permitir desabilitar coverage em MVP.
* Futuramente adicionar cache.

## 21. Resumo final

O **PR Quality Gate Dashboard** será uma aplicação web que conecta ao GitHub e analisa Pull Requests com foco em qualidade de software.

A primeira versão será simples:

```text
React Dashboard
+ FastAPI API
+ Postgres
+ SQLAlchemy
+ Alembic
+ LangChain
+ Docker Compose
```

O usuário poderá:

```text
conectar repositório
configurar regras
selecionar PR
rodar análise
ver PASS/FAIL
entender os problemas
publicar comentário/status no GitHub
```

O diferencial do projeto é combinar verificações objetivas com análise inteligente via LangChain. Assim, o sistema não apenas mostra que um PR falhou, mas explica por que falhou e sugere como corrigir.
